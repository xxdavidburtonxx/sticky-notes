# Electron Platform Integration — Sticky Notes

## Platform target

**Chosen: `macos-and-windows`.**

The reference (macOS Stickies) is a Mac-native idiom; macOS is first-class. Windows is the natural second target — the per-window note pattern translates fine. Linux is best-effort: the build will produce a `.deb`/`.rpm`/AppImage but no manual polish pass.

- Rejected `macos-only`: artificially narrow given the pattern is portable.
- Rejected `cross-platform` with full Linux polish: marginal user base, more testing burden than this v0 warrants.

To re-add full Linux later: smoke-test on at least one major DE (GNOME or KDE), audit the menu rendering and `window-all-closed` behavior, ship icons in the right PNG set.

## Per-platform behavior matrix

| Concern | macOS | Windows | Linux (best-effort) |
|---|---|---|---|
| Application menu visibility | Always shown in menu bar | Visible in note window | Visible in note window |
| `window-all-closed` action | Stay running (app remains, `Cmd+N` still creates) | Quit | Quit |
| `activate` event (dock click, no windows) | Create one fresh empty note window | N/A | N/A |
| Close button on a note | Closes window AND deletes record (Stickies semantics) | Same | Same |
| Quit shortcut | `Cmd+Q` | `Alt+F4` (window) / no app-level quit | `Ctrl+Q` |
| New Note shortcut | `Cmd+N` | `Ctrl+N` | `Ctrl+N` |
| Native title bar | Default | Default | Default (respect DE) |
| Dock/taskbar icon | Required `.icns` (deferred to ship) | Required `.ico` (deferred) | PNG set (deferred) |
| Tray icon | None | None | None |

The "Close button deletes record" behavior is unusual but matches the brief and Stickies. It is wired in the `multi-window-manager` capability via the `onClose` callback.

## Application menu

Standard 5-section menu (App on macOS / File / Edit / View / Window). The template's `src/main/menu.ts` already builds this; the `menu-actions` capability extends `File` with the `New Note` item.

Pseudo-code template (full code lives in the menu file):

```ts
const isMac = process.platform === 'darwin';
[
  ...(isMac ? [{ role: 'appMenu' /* effectively */ }] : []),
  {
    label: 'File',
    submenu: [
      {
        label: 'New Note',
        accelerator: 'CommandOrControl+N',
        click: createRecordAndOpenWindow,
      },
      { type: 'separator' },
      isMac ? { role: 'close' } : { role: 'quit' },
    ],
  },
  { role: 'editMenu' },
  { role: 'viewMenu' },
  ...(isMac ? [{ role: 'windowMenu' }] : [{ /* basic Window menu */ }]),
]
```

`role: 'editMenu'` gives the renderer's textareas standard cut/copy/paste/select-all wired. `role: 'windowMenu'` on macOS gives Bring All to Front (the multi-window pattern needs this).

## Keyboard shortcuts

| Action | Accelerator | Notes |
|---|---|---|
| New Note | `CommandOrControl+N` | Wired by `menu-actions`; works whether a note is focused, the app is focused with no notes, or via macOS dock. |
| Close (and delete) current note | `Cmd+W` (mac) / native close (Win/Linux) | Provided by `role: 'close'` in macOS File menu; provided by the OS window-close button on Win/Linux. The close handler in main fires the delete. |
| Quit | `Cmd+Q` (mac) / `Alt+F4` (Win) | Provided by `role: 'quit'`. |
| Standard text editing | `Cmd+Z`/`Cmd+X`/`Cmd+C`/`Cmd+V`/`Cmd+A` (or Ctrl on Win/Linux) | `role: 'editMenu'`. |
| DevTools (dev only) | `Cmd+Opt+I` / `Ctrl+Shift+I` | `role: 'toggleDevTools'`. Removed for shipped builds via fuses + production check. |

No global shortcuts (this isn't a quick-summon ambient app).

## Lifecycle handlers

```ts
// On launch:
app.whenReady().then(async () => {
  registerIpc();
  createApplicationMenu();
  const ids = await listAllRecordIds();
  if (ids.length === 0) {
    const id = await createRecord('');
    openRecordWindow(id, {}, deleteRecord);
  } else {
    for (const id of ids) openRecordWindow(id, {}, deleteRecord);
  }
});

// macOS dock-click with no windows: create a fresh empty note.
app.on('activate', async () => {
  if (BrowserWindow.getAllWindows().length === 0) {
    const id = await createRecord('');
    openRecordWindow(id, {}, deleteRecord);
  }
});

// Win/Linux: last note closed = quit. macOS: stay running.
app.on('window-all-closed', () => {
  if (process.platform !== 'darwin') app.quit();
});

// Each note window: on close, delete its record file.
// Wired by openRecordWindow's onClose callback (see multi-window-manager recipe).
```

`before-quit`: nothing to clean up beyond what's already debounced — record-store writes are atomic and synchronous-from-the-fs's-perspective; pending debounced writes in renderers are lost on quit, which matches Stickies (the user's last few keystrokes since the last 400ms tick are accepted as lost).

`second-instance`: not registered. The app doesn't have file associations or deep links; a second instance opening would be unusual but harmless. Single-instance lock is a v1 hardening, not required for v0.

## OS integration features

None used in v0. No file associations, no deep links, no custom protocols, no tray, no dock menu, no jump list, no vibrancy, no document-edited indicator (the close-deletes semantics make "edited" meaningless).

A nice-to-have for v1: a yellow window background tinted via macOS vibrancy (`vibrancy: 'under-window'`). Skipped for v0 to avoid adding `BrowserWindow` options that may interact with dragging/resizing on first attempt.

## Native modules

None. `files-on-disk` uses pure-Node `fs.promises`. No `better-sqlite3`, no `keytar`, no `sharp`. This means:
- No `electron-rebuild` step required.
- No per-platform CI matrix concern for native binary builds.
- Cross-compilation to Win/Linux from a Mac is straightforward.

If migration to SQLite ever happens, the native-rebuild story re-enters scope. For now: clean.

## Accessibility floor

- **Focus visible**: the `<textarea>` filling each note window inherits the browser default focus ring; do not override with `outline: none` without a replacement.
- **Keyboard reachable**: the textarea is the only interactive element in a note window; trivial.
- **No icon-only buttons**: there are no buttons in v0 — all actions are menu items, all menu items have text.
- **Color contrast**: black text on the optional yellow (#fff7a8) background passes WCAG AA easily (>15:1).
- **Native form input**: `<textarea>` is fully native; no custom replacement.

## Polish verification

Manual checks before declaring this dimension done:

1. Launch — first-launch: one empty note window appears (no flash-of-blank-window thanks to `ready-to-show`).
2. Type in the note; close the window; relaunch. The note is gone (file deleted on close).
3. Type in a note; quit (`Cmd+Q`); relaunch. The note's content is restored.
4. Open multiple notes via `Cmd+N`; quit; relaunch. All notes restored.
5. With multiple notes open, click `Window > Bring All to Front` (macOS). All notes raise to front.
6. With no notes open on macOS, click the dock icon. A fresh empty note appears.
7. With a note open on Windows, close it. App quits.
8. Trigger DevTools shortcut. DevTools panel opens (dev only); no errors in console.
9. `Cmd+W` on a focused note. Window closes (and record deletes).
10. `Cmd+Q` on macOS with notes open. App quits cleanly; relaunch shows the same notes.

If any check fails on any target platform, polish is not done for that platform. Linux checks 3, 4, 6, 7 are the floor; the rest are best-effort.
