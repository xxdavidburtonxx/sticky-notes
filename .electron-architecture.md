# Electron Architecture — Sticky Notes

## The decision in one paragraph

A multi-window, per-record Electron app where each "note" is its own small `BrowserWindow` bound to a UUID-keyed JSON record under `app.getPath('userData')/records/`. Main owns the canonical state, the window registry, and the record store; renderers are sandboxed textareas that talk through a narrow `contextBridge` for read/write/list. New notes are spawned via the `File > New Note` menu (`Cmd+N`); typing auto-saves debounced; closing a window deletes the underlying record file (Stickies semantics). On launch, all existing record files are reopened as windows; if there are none, one empty window is seeded. macOS keeps the app running with no open notes; Win/Linux quit on last-window-closed. There is no source repo; this is greenfield mode.

## Window model

**Chosen: `multi-window`.**

Each note is an independent `BrowserWindow`, sized small (default 320×320), peer to every other note window. The `Window` menu role on macOS gives "Bring All to Front" automatically.

- Rejected `single-window`: the brief is fundamentally per-document — the user expects N notes side by side, draggable independently. Single-window with internal panes wouldn't match the macOS Stickies feel.
- Rejected `menu-bar-tray`: notes need persistent OS-level windows the user can position; a tray popover would hide them on blur.
- Rejected `hybrid-tray-plus-main`: no quick-action surface implied by the brief; tray would be incidental complexity.

Source-of-decision signals: the brief explicitly names "each note is a small window" and "Cmd+N creates a new note window."

## Process boundaries

| Concern | Process |
|---|---|
| Window creation, menus, lifecycle | main |
| Record-file reads/writes (`fs`) | main |
| Window-to-record binding registry | main |
| `Cmd+N` menu action handler | main (menu callback creates record + window) |
| Textarea rendering, debounced change detection | renderer |
| Per-window in-memory body buffer | renderer |
| The contextBridge surface | preload |

Web preferences (every BrowserWindow):

```ts
contextIsolation: true
sandbox: true
nodeIntegration: false
```

Recorded explicitly so future deviations are conscious; none are anticipated.

The note-window URL carries a `?recordId=<uuid>` query param. Preload reads `globalThis.location.search` and exposes `recordId` to the renderer at bridge construction time. Main maps each `webContents` back to its record id for IPC validation — the renderer's claim to an id is never trusted.

## IPC surface

A small, capability-named contextBridge:

```ts
interface AppBridge {
  // Carried by preload from URL params; null if not a record window.
  recordId: string | null;

  // Per-record (validated against the caller's bound record id in main).
  records: {
    read(): Promise<{ id: string; body: string; createdAt: number; updatedAt: number } | null>;
    write(body: string): Promise<void>;
  };

  // Inherited from the template, kept narrow.
  openExternal(url: string): Promise<void>;
}
```

- `records.read` / `records.write` are **request/response** (`ipcMain.handle`).
- The renderer never passes its own id; main reads it from `BrowserWindow.fromWebContents(event.sender)` via the record-window registry. This is the path-traversal guard for record files.
- No subscription channels in v0 — single-writer-per-window means each renderer reads on mount and saves on its own; cross-window sync isn't required for the brief.
- "New Note" is a menu action, not an IPC channel. The menu callback runs in main directly.

Type sharing: a small `src/types/ipc.ts` is unnecessary at this size; the preload's `bridge` literal plus `global.d.ts` give the renderer types via the `AppBridge` export.

## Source binding

**Chosen: `none` (greenfield).**

There is no source repo. The Electron app *is* the source. Behavior is named in the **Behavior** section below; capability recipes (`multi-window-manager`, `record-store`, `menu-actions`) plus the `from-spec` UI seed write the code.

## Persistence model

**Chosen: `files-on-disk` with `record-shape: uuid-records`.**

- One JSON file per note: `<userData>/records/<uuid>.json` containing `{ id, body, createdAt, updatedAt }`.
- Atomic writes (`tmp + rename`) survive crashes mid-write.
- Listing notes = `readdir` of the directory, filter to `*.json` whose stem is a valid UUID.
- Deletion = `fs.unlink` of the record file, fired from the window's `close` handler.
- Migrations: schema is one flat object; if it grows, add an optional field and read tolerantly. No version field needed at v0.

Rejected `sqlite`: overkill for an open-ended list of records with no relations and no queries. SQLite's native-module rebuild story is real cost; a flat directory of JSON is inspectable, backupable, and trivially syncable.

Rejected `none`: the brief explicitly requires saved notes that survive launch.

Rejected `remote`: no backend; this is a fully local app.

## Behavior

The 5 user-visible behaviors that drive the rest of the build:

1. **Launch behavior.** On startup, scan `<userData>/records/`. If empty, create one empty record and open its window. Otherwise, open one window per existing record (restoring the previous session).
2. **`Cmd+N` creates a new note window.** Wired through `File > New Note` in the application menu. Creates a record with empty body, opens its window.
3. **Typing in a note auto-saves.** Renderer debounces 400ms; on tick, calls `window.app.records.write(body)`. Main re-writes the record file atomically.
4. **Closing a note window deletes its record file.** Stickies semantics. Wired via `BrowserWindow.on('close', ...)` in main, calling the record-store's `deleteRecord(id)`.
5. **Last-window-closed behavior is platform-conventional.** macOS: app stays running, dock icon present, `Cmd+N` from the menu still works. Win/Linux: app quits.

Data shape:

```ts
type Record = {
  id: string;        // UUID v4
  body: string;      // textarea content
  createdAt: number; // ms epoch
  updatedAt: number; // ms epoch
};
```

IPC namespace implied by these behaviors:

- `record:read` (per-record-window, id resolved from sender)
- `record:write` (per-record-window, id resolved from sender)
- Menu-driven `createRecord + openRecordWindow` (no IPC channel — runs in main)
- `BrowserWindow.on('close')` → `deleteRecord` (no IPC channel — main internal)

Reference points: macOS Stickies (specifically: close-deletes, peer windows, tiny chrome, yellow background optional but iconic).

## Consequences and follow-on decisions

What this architecture forecloses:
- **No "list all notes" view in v0.** Each note is its own window; there is no overview window. Adding one later is straightforward (a non-record-bound main window with an `ipcMain.handle('records:list')` channel).
- **No cross-window real-time sync.** Each window owns its body. If two windows somehow bind to the same id (they won't, by registry), edits would clobber each other. The `multi-window-manager` recipe enforces id uniqueness via `Map<string, BrowserWindow>`.
- **No undo across notes.** The textarea has its own native undo per window; that's sufficient for v0.

What it makes easy:
- Adding a "list view" main window later: a single new IPC handler over the existing record store.
- Adding window-bounds persistence: extend the `Record` shape with `bounds: { x, y, width, height }` (the `multi-window-manager` recipe already shows the wiring).
- Replacing `files-on-disk` with `sqlite` later: the IPC surface stays identical; only the record-store implementation changes.

Downstream decisions now constrained:
- **Platform integration:** menu must include `File > New Note  Cmd+N`; lifecycle on macOS must keep app alive with no open windows (the platform memo's job).
- **Delivery:** standard hardened defaults; no special signing or auto-update needs in v0.
