# Electron Delivery — Sticky Notes

## Security posture

Standard hardened defaults inherited from the template:

```ts
// Every BrowserWindow:
webPreferences: {
  contextIsolation: true,
  sandbox: true,
  nodeIntegration: false,
  webSecurity: true,
  preload: path.join(__dirname, 'preload.js'),
}
```

No relaxations. The `record-id` query string is read by preload at construction time and exposed read-only — preload is the only code with `ipcRenderer`, and main validates every per-record IPC call against the registered record-id of the calling `webContents` (not the renderer's claim).

### Content Security Policy

**Deferred to ship-time.** v0 ships no CSP because Vite HMR breaks under strict CSP and the renderer is so simple (a textarea, no remote calls) that the attack surface is minimal during development. When a packaged build is produced, install a strict CSP via `webRequest.onHeadersReceived` per the `csp-production` capability:

```
default-src 'self';
script-src 'self';
style-src 'self' 'unsafe-inline';
img-src 'self' data: blob:;
font-src 'self' data:;
connect-src 'self';
base-uri 'none';
form-action 'none';
frame-ancestors 'none';
```

`connect-src 'self'` is correct — the renderer makes no network calls. If iteration ever adds them, narrow this allowlist to the specific origin.

### Web content navigation hardening

Inherited from the template's `src/main/external-links.ts`:
- `setWindowOpenHandler` denies popups and routes external `http(s):` links to `shell.openExternal`.
- `will-navigate` redirects external navigations to the system browser.

`will-attach-webview` not currently registered — but the architecture forbids `<webview>` use entirely, so a hardening pass should add an explicit `event.preventDefault()` on every window's `will-attach-webview` for defense in depth. **Action item for v1.**

### safeStorage

Not used in v0 — there are no secrets. Notes are user-controlled plain text; encrypting them would be theatre, not security (the user can read them; the OS file-system permissions are the boundary).

### Permissions handlers

Not configured in v0. Renderers don't request camera, mic, geolocation, notifications, or anything else. If a future iteration adds notifications, install `setPermissionRequestHandler` with a narrow allowlist.

## Packaging stack

**Chosen: `electron-forge` + `@electron-forge/plugin-vite` + TypeScript + React 18.**

This is the template default; no deviation. Makers configured for all targets:

- macOS: `MakerDMG` (preferred for direct download) + `MakerZIP` (used by Squirrel.Mac for auto-update).
- Windows: `MakerSquirrel` (produces `.exe` installer + auto-update channel).

`asar: true` packs app code into a single archive. `extraResource` is empty — there are no extra binaries or static assets to ship.

`AutoUnpackNativesPlugin` is in the plugin list; harmless even though no native modules are used (idle until needed).

`rebuildConfig: {}` likewise harmless; no native modules to rebuild.

Nothing is expected on PATH at runtime — the app is fully self-contained.

## Per-platform signing

### macOS — Developer ID + notarization

**Deferred to v1.** v0 ships unsigned for personal/dev use. To configure when signing infrastructure is in place:

```ts
packagerConfig: {
  osxSign: {
    identity: 'Developer ID Application: <Org> (<TEAM_ID>)',
    optionsForFile: () => ({
      entitlements: 'build/entitlements.mac.plist',
      hardenedRuntime: true,
    }),
  },
  osxNotarize: {
    appleId: process.env.APPLE_ID,
    appleIdPassword: process.env.APPLE_APP_PASSWORD,
    teamId: process.env.APPLE_TEAM_ID,
  },
}
```

Entitlements file (`build/entitlements.mac.plist`) needs to declare:
- `com.apple.security.cs.allow-jit` — Electron's V8 engine.

Network entitlement (`com.apple.security.network.client`) NOT needed — the renderer makes no network calls.

### Windows — Authenticode

**Deferred to v1.** v0 ships unsigned; users will see SmartScreen "Unknown publisher" warning. Acceptable for personal/dev distribution; not for general release.

### Linux — GPG

Skipped. v0 ships unsigned `.deb`/`.rpm`/AppImage; users who care can verify SHA256 from a hosted manifest.

## Distribution channel

**Chosen: `direct-download` (v0).**

Hosting target: GitHub Releases (or whatever the user prefers). SHA256 checksums alongside the artifacts.

Rationale: v0 is for personal use and a small audience. Direct download has zero infrastructure cost, simplest release process, and acceptable user friction. Auto-update can be added later without re-architecting.

To upgrade to `direct-with-autoupdate`:
1. Add `update-electron-app` per the `auto-update` capability recipe.
2. Wire it production-only via `app.isPackaged`.
3. Use `update.electronjs.org` (free, GitHub-Releases-backed) or self-host.

## Auto-update strategy

Not configured in v0. See above for the v1 path.

When configured, recommended:
- **Tool**: `update-electron-app` (default for Forge).
- **Feed**: `update.electronjs.org` for free GitHub Releases hosting.
- **Cadence**: default — check at startup, prompt on next launch if newer version found.
- **Rollback**: ship a hotfix; users on the bad version skip past it. No rollback protocol planned.
- **Version skew**: schema-tolerant reads (`record.bounds` is optional) mean any v0.x → v0.y update is forward-compatible.

## Ship verification

1. **Fresh-machine install** — copy the `.dmg`/`.exe` to a machine with no developer tools. Install. Launch. Verify:
   - First-launch shows one empty note.
   - `Cmd+N` creates a second window.
   - Type, close, relaunch — note is gone (close-deletes worked).
   - Type, quit, relaunch — note is restored.
2. **Updater works** (deferred to v1).
3. **Uninstall is clean** — uninstalling on macOS leaves `~/Library/Application Support/Sticky Notes/` (intended; user data may be deliberate). Document in release notes.
4. **Crash reports** — n/a for v0.
5. **Signed and notarized** — deferred.
